## 1. 관리자 역할

### 1.1 시스템 관리자의 책임

TaskFlow 시스템 관리자는 다음과 같은 책임을 가집니다:

| 책임 영역 | 설명 |
|----------|------|
| **사용자 관리** | 사용자 계정 생성, 수정, 비활성화 및 비밀번호 정책 관리 |
| **조직 구조 관리** | 부서 및 그룹 구조 설정 및 유지보수 |
| **시스템 설정** | 속성 정의, 코드 값, 템플릿 등 시스템 기본 설정 관리 |
| **보안 관리** | 접근 권한, 인증 설정, 보안 정책 유지 |
| **데이터 관리** | 백업, 복원, 데이터 정합성 유지 |
| **시스템 모니터링** | 서버 상태, 로그, 성능 모니터링 |
| **문제 해결** | 사용자 문의 대응, 장애 처리 |

### 1.2 관리자 권한

현재 버전에서는 인증된 모든 사용자가 관리 기능에 접근할 수 있습니다. 향후 버전에서는 ADMIN 역할로 제한될 예정입니다.

**API 권한:**
- `@PreAuthorize("isAuthenticated()")` - 모든 인증된 사용자
- 향후 `@PreAuthorize("hasRole('ADMIN')")` 으로 변경 예정

### 1.3 관리 메뉴 접근

TaskFlow의 관리 기능은 다음 메뉴를 통해 접근합니다:

```
사이드바 메뉴:
├── 사용자 등록 메뉴
├── 공유 사용자 등록 메뉴
├── 부서 관리 메뉴
├── 그룹 관리 메뉴
├── 작업 등록 메뉴 (템플릿)
└── 이력관리 메뉴 (감사 로그)
```

## 2. 사용자 관리

### 2.1 사용자 등록

**API 엔드포인트:** `POST /api/users`

**등록 절차:**

1. 사용자 등록 메뉴 접근
2. 필수 정보 입력:
   - **사용자 이름**: 표시될 이름
   - **아이디**: 로그인 ID (영문+숫자, 4~20자)
   - **패스워드**: 최소 8자, 영문+숫자+특수문자
   - **패스워드 확인**: 재입력
   - **이메일**: 사용자 이메일 주소
   - **부서**: 소속 부서 선택 (선택사항)

3. 아이디 중복 확인 (`GET /api/users/check-username?username=...`)
4. 등록 완료

**요청 예시:**
```json
{
  "username": "hong123",
  "password": "Pass123!@#",
  "name": "홍길동",
  "email": "hong@example.com",
  "departmentId": 1
}
```

### 2.2 사용자 정보 수정

**API 엔드포인트:** `PUT /api/users/{id}`

**수정 가능 항목:**
- 사용자 이름
- 이메일
- 부서
- 사용 여부 (활성화/비활성화)

**요청 예시:**
```json
{
  "name": "홍길동",
  "email": "hong.updated@example.com",
  "departmentId": 2,
  "useYn": "Y"
}
```

### 2.3 비밀번호 변경

**API 엔드포인트:** `PUT /api/users/{id}/password`

**변경 절차:**
1. 대상 사용자 선택
2. 새 비밀번호 입력
3. 비밀번호 정책 검증
4. 변경 완료

**요청 예시:**
```json
{
  "newPassword": "NewPass123!@#",
  "confirmPassword": "NewPass123!@#"
}
```

### 2.4 비밀번호 정책

시스템에 적용된 비밀번호 정책은 다음과 같습니다:

| 정책 | 요구사항 |
|------|----------|
| **최소 길이** | 8자 이상 |
| **영문 대문자** | 1개 이상 포함 |
| **영문 소문자** | 1개 이상 포함 |
| **숫자** | 1개 이상 포함 |
| **특수문자** | 1개 이상 포함 (!@#$%^&*) |
| **암호화 방식** | BCrypt (strength: 10) |

**정책 위반 시:** 등록/변경이 거부되며 오류 메시지가 표시됩니다.

### 2.5 사용자 비활성화/활성화

사용자를 삭제하는 대신 비활성화하여 데이터를 보존할 수 있습니다:

**비활성화:**
- 사용자 수정 화면에서 `useYn` 을 `N` 으로 설정
- 비활성화된 사용자는 로그인 불가
- 기존 데이터는 모두 유지

**활성화:**
- `useYn` 을 `Y` 로 설정하여 재활성화

### 2.6 사용자 삭제

**API 엔드포인트:** `DELETE /api/users/{id}`

**주의사항:**
- 사용자 삭제는 물리적 삭제가 아닌 논리적 삭제입니다
- 삭제된 사용자의 작업 이력은 유지됩니다
- 복구가 필요한 경우 데이터베이스에서 직접 처리해야 합니다

### 2.7 사용자 목록 조회

**API 엔드포인트:** `GET /api/users`

**필터링 옵션:**
```
GET /api/users?page=0&size=20&keyword=홍길동&departmentId=1&useYn=Y
```

| 파라미터 | 설명 | 예시 |
|----------|------|------|
| page | 페이지 번호 (0부터 시작) | 0 |
| size | 페이지 크기 | 20 |
| keyword | 검색 키워드 (이름/아이디) | 홍길동 |
| departmentId | 부서 필터 | 1 |
| useYn | 사용 여부 필터 | Y |

### 2.8 부서별 사용자 조회

**API 엔드포인트:** `GET /api/departments/{id}/users`

특정 부서에 소속된 모든 사용자를 조회합니다.

## 3. 부서 관리

### 3.1 부서 트리 구조 이해

TaskFlow의 부서는 계층 구조(트리)를 지원합니다:

```
경영지원본부 (최상위)
├── 인사팀 (하위)
├── 총무팀 (하위)
└── 재무팀 (하위)

개발본부 (최상위)
├── 개발1팀 (하위)
├── 개발2팀 (하위)
└── QA팀 (하위)
```

**특징:**
- 최상위 부서는 `PARENT_DEPARTMENT_ID` 가 NULL
- 하위 부서는 상위 부서의 ID를 참조
- 무제한 깊이의 계층 구조 지원

### 3.2 부서 생성

**API 엔드포인트:** `POST /api/departments`

**필수 정보:**
- **부서 코드**: 고유 식별자 (예: DEV, HR, MKT)
- **부서명**: 표시될 부서 이름
- **상위 부서**: 계층 구조 설정 (선택사항)
- **정렬 순서**: 표시 순서 (선택사항)
- **사용 여부**: 활성화 여부 (기본: Y)

**요청 예시:**
```json
{
  "departmentCode": "DEV",
  "departmentName": "개발팀",
  "parentDepartmentId": null,
  "sortOrder": 1,
  "useYn": "Y"
}
```

**부서 코드 중복 확인:**
```
GET /api/departments/check-code?code=DEV
```

### 3.3 부서 수정

**API 엔드포인트:** `PUT /api/departments/{id}`

**수정 가능 항목:**
- 부서명
- 상위 부서 (계층 구조 변경)
- 정렬 순서
- 사용 여부

**주의사항:**
- 부서 코드는 수정 불가
- 상위 부서 변경 시 순환 참조 주의

### 3.4 부서 순서 변경

**API 엔드포인트:** `PUT /api/departments/{id}/order`

**요청 예시:**
```json
{
  "sortOrder": 5
}
```

**드래그 앤 드롭 지원:**
- 프론트엔드에서 드래그 앤 드롭으로 순서 변경 가능
- 변경된 순서는 자동으로 API 호출

### 3.5 부서 삭제

**API 엔드포인트:** `DELETE /api/departments/{id}`

**삭제 전 확인사항:**
1. **하위 부서 존재 여부**
   - 하위 부서가 있는 경우 경고 표시
   - 하위 부서를 먼저 삭제하거나 다른 부서로 이동 필요

2. **소속 사용자 존재 여부**
   - 해당 부서에 소속된 사용자가 있는 경우 경고
   - 사용자를 다른 부서로 이동 후 삭제 권장

**권장 절차:**
1. 비활성화(`useYn=N`) 후 일정 기간 유지
2. 문제 없으면 물리적 삭제

### 3.6 부서 목록 조회

**트리 구조 조회:**
```
GET /api/departments?useYn=Y
```

**평면 구조 조회 (SELECT 박스용):**
```
GET /api/departments/flat?useYn=Y
```

**하위 부서 조회:**
```
GET /api/departments/{id}/children
```

### 3.7 부서 관리 베스트 프랙티스

1. **부서 코드 네이밍 규칙**
   - 영문 대문자 사용 (예: DEV, HR, MKT)
   - 3~10자 이내
   - 의미 있는 약어 사용

2. **계층 구조 설계**
   - 3~4단계 이내로 제한 권장
   - 너무 깊은 계층은 관리 복잡도 증가

3. **정렬 순서**
   - 10 단위로 증가 권장 (10, 20, 30...)
   - 중간 삽입 시 유연성 확보

## 4. 그룹 관리

### 4.1 그룹 vs 부서 차이점

| 구분 | 부서 | 그룹 |
|------|------|------|
| **구조** | 조직도 기반의 계층 구조 | 평면 구조 |
| **소속** | 1인 1부서 | 1인 다중 그룹 가능 |
| **용도** | 고정된 조직 소속 | 프로젝트/업무 팀 구성 |
| **예시** | 개발팀, 인사팀 | 신규 서비스 개발 TF, 보안 점검 그룹 |

### 4.2 그룹 생성

**API 엔드포인트:** `POST /api/groups`

**필수 정보:**
- **그룹 코드**: 고유 식별자
- **그룹명**: 표시될 그룹 이름
- **그룹 설명**: 그룹 용도 설명 (선택사항)
- **그룹 색상**: 표시용 색상 (선택사항)
- **정렬 순서**: 표시 순서 (선택사항)
- **사용 여부**: 활성화 여부 (기본: Y)

**요청 예시:**
```json
{
  "groupCode": "PROJ_NEW_SERVICE",
  "groupName": "신규 서비스 개발",
  "description": "2025년 신규 서비스 개발 프로젝트",
  "colorCode": "#FF5733",
  "sortOrder": 1,
  "useYn": "Y"
}
```

**그룹 코드 중복 확인:**
```
GET /api/groups/check-code?code=PROJ_NEW_SERVICE
```

### 4.3 그룹 수정

**API 엔드포인트:** `PUT /api/groups/{id}`

**수정 가능 항목:**
- 그룹명
- 설명
- 색상
- 정렬 순서
- 사용 여부

### 4.4 그룹 멤버 관리

**멤버 목록 조회:**
```
GET /api/groups/{id}/members
```

**멤버 추가:**
```
POST /api/groups/{id}/members

{
  "userId": 5
}
```

**멤버 제거:**
```
DELETE /api/groups/{id}/members/{userId}
```

**사용자별 소속 그룹 조회:**
```
GET /api/users/{userId}/groups
```

### 4.5 그룹 삭제

**API 엔드포인트:** `DELETE /api/groups/{id}`

**삭제 전 확인사항:**
1. **그룹 멤버 존재 여부**
   - 모든 멤버를 먼저 제거 권장

2. **업무 아이템 연결 여부**
   - 해당 그룹에 할당된 업무가 있는지 확인
   - 업무를 다른 그룹으로 변경 후 삭제

### 4.6 그룹 활용 예시

**프로젝트 그룹:**
```
- 신규 서비스 개발
- 시스템 리뉴얼 프로젝트
- AI 도입 TF
```

**업무 그룹:**
```
- 주간 보고
- 월간 회의
- 분기 계획
```

**TF 그룹:**
```
- 보안 점검 TF
- 비용 절감 TF
- 프로세스 개선 TF
```

### 4.7 그룹 관리 베스트 프랙티스

1. **그룹 네이밍**
   - 목적이 명확하게 드러나도록 작성
   - 예: "PROJ_" (프로젝트), "TF_" (TF), "TEAM_" (팀)

2. **색상 코딩**
   - 프로젝트별 색상 통일
   - 중요도/긴급도에 따른 색상 구분

3. **주기적인 정리**
   - 종료된 프로젝트 그룹은 비활성화
   - 6개월~1년 단위로 그룹 정리

## 5. 보드 관리

### 5.1 보드 개념

보드는 업무 아이템을 담는 컨테이너입니다:
- 사용자는 여러 개의 보드를 생성 가능
- 각 보드는 독립적인 속성 정의를 가짐
- 보드별로 다른 사용자와 공유 가능

### 5.2 보드 목록 조회

**전체 접근 가능한 보드 (소유 + 공유):**
```
GET /api/boards?useYn=Y
```

**소유한 보드만:**
```
GET /api/boards?owned=true&useYn=Y
```

**소유/공유 분리 조회:**
```
GET /api/boards/list
```

응답 예시:
```json
{
  "ownedBoards": [...],
  "sharedBoards": [...]
}
```

### 5.3 보드 생성

**API 엔드포인트:** `POST /api/boards`

**요청 예시:**
```json
{
  "boardName": "2025년 1분기 업무",
  "description": "2025년 1분기 주요 업무 관리",
  "useYn": "Y"
}
```

### 5.4 보드 공유 관리

**공유 사용자 목록 조회:**
```
GET /api/boards/{id}/shares
```

**공유 사용자 추가:**
```
POST /api/boards/{id}/shares

{
  "userId": 10,
  "permission": "EDIT"
}
```

**권한 종류:**
- `EDIT`: 편집 권한 (기본)
- `VIEW`: 조회 전용 (향후 확장)

**공유 권한 변경:**
```
PUT /api/boards/{id}/shares/{userId}

{
  "permission": "VIEW"
}
```

**공유 사용자 제거:**
```
DELETE /api/boards/{id}/shares/{userId}
```

### 5.5 보드 소유권 이전

**API 엔드포인트:** `PUT /api/boards/{id}/transfer-ownership`

**사용 시나리오:**
- 담당자 퇴사/이직
- 업무 인수인계

**요청 예시:**
```json
{
  "targetUserId": 15
}
```

**주의사항:**
- 보드명이 자동으로 "보드이관"으로 변경됨
- 소유자만 이관 가능
- 이관 후 복구 불가

### 5.6 보드 삭제 (업무 이관 포함)

**API 엔드포인트:** `DELETE /api/boards/{id}/with-transfer`

**이관 미리보기:**
```
GET /api/boards/{id}/transfer-preview
```

응답 예시:
```json
{
  "totalItems": 25,
  "activeItems": 15,
  "completedItems": 8,
  "deletedItems": 2
}
```

**삭제 요청:**

**파라미터 설명:**

## 9. 백업 및 복원

### 9.1 데이터베이스 백업

**수동 백업:**

```bash
# 전체 데이터베이스 백업
docker exec taskflow-mysql mysqldump \
  -u root -p${MYSQL_ROOT_PASSWORD} \
  taskflow > backup_$(date +%Y%m%d_%H%M%S).sql

# 압축 백업
docker exec taskflow-mysql mysqldump \
  -u root -p${MYSQL_ROOT_PASSWORD} \
  taskflow | gzip > backup_$(date +%Y%m%d_%H%M%S).sql.gz
```

**특정 테이블만 백업:**
```bash
docker exec taskflow-mysql mysqldump \
  -u root -p${MYSQL_ROOT_PASSWORD} \
  taskflow TB_USER TB_DEPARTMENT TB_GROUP \
  > backup_master_tables.sql
```

**스키마만 백업 (데이터 제외):**
```bash
docker exec taskflow-mysql mysqldump \
  -u root -p${MYSQL_ROOT_PASSWORD} \
  --no-data taskflow > schema_only.sql
```

### 9.2 자동 백업 스크립트

**backup.sh 예시:**
```bash
#!/bin/bash

BACKUP_DIR="/backup/taskflow"
MYSQL_ROOT_PASSWORD="your_password"
DATE=$(date +%Y%m%d_%H%M%S)

# 백업 디렉토리 생성
mkdir -p ${BACKUP_DIR}

# MySQL 백업
docker exec taskflow-mysql mysqldump \
  -u root -p${MYSQL_ROOT_PASSWORD} \
  taskflow | gzip > ${BACKUP_DIR}/db_${DATE}.sql.gz

# 7일 이상된 백업 파일 삭제
find ${BACKUP_DIR} -name "db_*.sql.gz" -mtime +7 -delete

echo "Backup completed: db_${DATE}.sql.gz"
```

**크론탭 등록 (매일 새벽 2시):**
```bash
crontab -e

0 2 * * * /path/to/backup.sh >> /var/log/taskflow-backup.log 2>&1
```

### 9.5 전체 시스템 백업

**백업 항목:**
1. MySQL 데이터베이스
2. 업로드 파일 (file_uploads 볼륨)
3. 환경 설정 파일 (.env)
4. 커스텀 설정 파일

**통합 백업 스크립트:**
```bash
#!/bin/bash

BACKUP_ROOT="/backup/taskflow-full"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="${BACKUP_ROOT}/${DATE}"

mkdir -p ${BACKUP_DIR}

# 1. MySQL 백업
docker exec taskflow-mysql mysqldump -u root -p${MYSQL_ROOT_PASSWORD} taskflow | \
  gzip > ${BACKUP_DIR}/database.sql.gz

# 2. 파일 볼륨 백업
docker run --rm \
  -v taskflow_file_uploads:/source \
  -v ${BACKUP_DIR}:/backup \
  alpine tar czf /backup/files.tar.gz -C /source .

# 3. 환경 설정 백업
cp .env ${BACKUP_DIR}/env.backup

# 4. 백업 압축
cd ${BACKUP_ROOT}
tar czf taskflow_full_${DATE}.tar.gz ${DATE}
rm -rf ${DATE}

echo "Full backup completed: taskflow_full_${DATE}.tar.gz"
```

## 10. 보안 관리

### 10.1 JWT 토큰 설정

**application.yml 설정:**

```yaml
jwt:
  secret: ${JWT_SECRET}              # Base64 인코딩된 256비트 이상
  access-token-validity: 1800000     # 30분 (밀리초)
  refresh-token-validity: 604800000  # 7일 (밀리초)
```

**JWT 시크릿 키 생성:**

```bash
# Linux/macOS
openssl rand -base64 32

# Windows PowerShell
[Convert]::ToBase64String((1..32 | ForEach-Object { Get-Random -Maximum 256 }) -as [byte[]])
```

**토큰 저장 위치:**
- **Access Token**: localStorage (클라이언트)
- **Refresh Token**: httpOnly Cookie (보안)

**토큰 갱신 플로우:**
1. Access Token 만료 시 자동 감지
2. Refresh Token으로 갱신 요청
3. 새 Access Token 발급
4. Refresh Token도 만료 시 재로그인 필요

### 10.2 CORS 설정

**application.yml 설정:**

```yaml
cors:
  allowed-origins: ${CORS_ALLOWED_ORIGINS}
  max-age: 3600
```

**.env 파일 설정:**

```env
# 단일 Origin
CORS_ALLOWED_ORIGINS=http://localhost:3000

# 다중 Origin (쉼표로 구분)
CORS_ALLOWED_ORIGINS=http://localhost:3000,http://localhost:5173

# 프로덕션
CORS_ALLOWED_ORIGINS=https://taskflow.example.com
```

**주의사항:**
- 프론트엔드 포트 변경 시 반드시 CORS도 함께 변경
- `*` (모든 Origin 허용)는 절대 사용 금지
- HTTPS 사용 시 반드시 https:// URL 사용

### 10.3 비밀번호 정책 적용

**백엔드 검증:**

서비스 레벨에서 비밀번호 정책 검증:
```java
- 최소 8자 이상
- 영문 대문자 1개 이상
- 영문 소문자 1개 이상
- 숫자 1개 이상
- 특수문자 1개 이상 (!@#$%^&*)
```

**암호화:**
- BCrypt 알고리즘 사용
- Strength: 10

**사용자에게 안내:**
- 비밀번호 정책을 회원가입/변경 화면에 명시
- 정책 위반 시 구체적인 오류 메시지 표시

### 10.5 SQL Injection 방지

TaskFlow는 MyBatis를 사용하여 SQL Injection을 방지합니다:

**안전한 쿼리 작성:**
```xml
<!-- 안전 (파라미터 바인딩) -->
<select id="findByUsername" resultType="User">
  SELECT * FROM TB_USER
  WHERE USERNAME = #{username}
</select>

<!-- 위험 (문자열 연결) - 사용 금지 -->
<select id="findByUsername" resultType="User">
  SELECT * FROM TB_USER
  WHERE USERNAME = '${username}'
</select>
```

**원칙:**
- `#{}` 사용 (PreparedStatement)
- `${}` 절대 사용 금지 (단, ORDER BY 등 예외적 상황만)

2. 허용된 확장자만 업로드
3. MIME 타입 검증
4. 파일명 검증 (경로 탐색 방지)

### 10.7 접근 제어

**현재 구현:**
- `@PreAuthorize("isAuthenticated()")` - 인증된 사용자만

**향후 확장:**
```java
@PreAuthorize("hasRole('ADMIN')")      // 관리자만
@PreAuthorize("hasRole('USER')")       // 일반 사용자
@PreAuthorize("hasAnyRole('ADMIN','MANAGER')") // 다중 역할
```

## 11. 문제 해결 가이드

### 11.1 로그인 문제

**증상 1: "아이디 또는 비밀번호가 일치하지 않습니다"**

**원인:**
- 잘못된 아이디/비밀번호
- 비활성화된 계정 (USE_YN = 'N')
- 데이터베이스 연결 문제

**해결:**
```sql
-- 1. 사용자 존재 확인
SELECT USER_ID, USERNAME, NAME, USE_YN
FROM TB_USER
WHERE USERNAME = 'admin';

-- 2. 계정 활성화
UPDATE TB_USER
SET USE_YN = 'Y'
WHERE USERNAME = 'admin';

-- 3. 비밀번호 초기화 (BCrypt로 암호화된 값)
UPDATE TB_USER
SET PASSWORD = '$2a$10$...'  -- 새 비밀번호
WHERE USERNAME = 'admin';
```

**증상 2: "토큰이 만료되었습니다"**

**원인:**
- Access Token 만료 (30분)
- Refresh Token 만료 (7일)

**해결:**

- 재로그인

**증상 3: CORS 에러**

**원인:**

- 프론트엔드 URL과 허용 Origin 불일치
